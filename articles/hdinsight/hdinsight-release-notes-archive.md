## <a name="summary"></a>Összefoglalás

Az Azure HDInsight az egyik legnépszerűbb szolgáltatás a nagyvállalati ügyfelek körében a nyílt forráskódú Apache Hadoop és az Apache Spark-elemzések hez az Azure-ban.

### <a name="new-features"></a>Új funkciók

#### <a name="service-tags"></a>Szolgáltatáscímkék
A szolgáltatáscímkék leegyszerűsítik az Azure virtuális gépek és az Azure virtuális hálózatok biztonságát azáltal, hogy lehetővé teszik az Azure-szolgáltatások hoz való hálózati hozzáférés egyszerű korlátozását. A hálózati biztonsági csoport (NSG) szabályaiban szolgáltatáscímkék használatával engedélyezheti vagy megtagadhatja a forgalmat egy adott Azure-szolgáltatáshoz globálisan vagy Azure-régiónként. Az Azure biztosítja az egyes címkék alapjául szolgáló IP-címek karbantartását. A hálózati biztonsági csoportok HDInsight szolgáltatáscímkéi az állapot- és felügyeleti szolgáltatások IP-címcsoportjai. Ezek a csoportok segítenek minimalizálni a biztonsági szabályok létrehozásának összetettségét. A HDInsight-ügyfelek az Azure Portalon, a PowerShellen és a REST API-n keresztül engedélyezhetik a szolgáltatáscímkét. További információ: [Network security group (NSG) service tags for Azure HDInsight.](https://docs.microsoft.com/azure/hdinsight/hdinsight-service-tags)

#### <a name="custom-ambari-db"></a>Egyéni Ambari-adatbázis
A HDInsight mostantól lehetővé teszi, hogy saját SQL DB-t használjon az Apache Ambari számára. Ezt az egyéni Ambari ADATBÁZISt az Azure Portalon vagy az erőforrás-kezelő sablonon keresztül konfigurálhatja.  Ez a funkció lehetővé teszi, hogy válassza ki a megfelelő SQL DB a feldolgozási és kapacitási igényeket. Könnyedén frissíthet is, hogy megfeleljen az üzleti növekedés követelményeinek. További információt a [HDInsight-fürtök beállítása egyéni Ambari ADATBÁZISsal](hdinsight-custom-ambari-db.md)című témakörben talál.

#### <a name="transport-layer-security-tls-12-enforcement"></a>Közlekedési réteg biztonsága (TLS) 1.2 kényszerítés
A Transport Layer Security (TLS) és a Secure Sockets Layer (SSL) olyan kriptográfiai protokollok, amelyek számítógépes hálózaton keresztül biztosítják a kommunikáció biztonságát. További információ: [Transport Layer Security](https://en.wikipedia.org/wiki/Transport_Layer_Security#SSL_1.0.2C_2.0_and_3.0). Míg az Azure HDInsight-fürtök elfogadják a TLS 1.2-es kapcsolatokat nyilvános HTTPS-végpontokon, a TLS 1.1 továbbra is támogatott a régebbi ügyfelekkel való visszamenőleges kompatibilitás érdekében.

A következő kiadástól kezdve engedélyezheti és konfigurálhatja az új HDInsight-fürtöket úgy, hogy csak TLS 1.2-es kapcsolatokat fogadjanak el. 

Az év későbbi részében, kezdve 6/30/2020, az Azure HDInsight kényszeríti TLS 1.2 vagy újabb verziók az összes HTTPS-kapcsolatok. Azt javasoljuk, hogy győződjön meg arról, hogy minden ügyfele készen áll a TLS 1.2-es vagy újabb verziók kezelésére.

Jelenleg a HIB csak az Azure Resource Management (ARM) sablonon keresztül telepített fürtök esetében érhető el.

A Kafka Rest API-proxy egy kattintással biztosítja a magas rendelkezésre állású REST-proxy kafka-fürttel való üzembe helyezését a biztonságos AAD-engedélyezés és az OAuth protokoll segítségével. 

#### <a name="auto-scale"></a>Automatikus méretezés

Az Azure HDInsight automatikus skálázása mostantól általánosan elérhető az Összes régióban az Apache Spark és a Hadoop fürttípusokhoz. Ez a funkció lehetővé teszi a big data-elemzési számítási feladatok költséghatékonyabb és hatékonyabb kezelését. Most már optimalizálhatja a HDInsight-fürtök használatát, és csak azért kell fizetnie, amire szüksége van.

A követelményektől függően választhat a terhelésalapú és az ütemezésalapú automatikus skálázás között. A terhelésalapú automatikus skálázás az aktuális erőforrásigények alapján skálázhatja a fürt méretet, míg az ütemezésalapú automatikus skálázás egy előre meghatározott ütemezés alapján módosíthatja a fürt méretét. 

A HBase és az LLAP számítási feladatok automatikus skálázási támogatása is nyilvános előzetes verzió. További információt az [Azure HDInsight-fürtök automatikus méretezése című témakörben talál.](https://docs.microsoft.com/azure/hdinsight/hdinsight-autoscale-clusters)

### <a name="known-issues"></a>Ismert problémák

#### <a name="replication-is-broken-for-secure-hbase-with-default-installation"></a>A biztonságos HBase replikációja megszakadt az alapértelmezett telepítéssel

A HDInsight 4.0 esetén tegye a következőket:

1. Fürtközi kommunikáció engedélyezése.
1. Jelentkezzen be az aktív csomópontba.
1. Töltsön le egy parancsfájlt a replikáció engedélyezéséhez a következő paranccsal:

    ```
    sudo wget https://raw.githubusercontent.com/Azure/hbase-utils/master/replication/hdi_enable_replication.sh
    ```
1. Írja be a `sudo kinit <domainuser>` parancsot.
1. Írja be a következő parancsot a parancsfájl futtatásához:

    ```
    sudo bash hdi_enable_replication.sh -m <hn0> -s <srclusterdns> -d <dstclusterdns> -sp <srcclusterpasswd> -dp <dstclusterpasswd> -copydata
    ```
A HDInsight 3.6 esetén tegye a következőket:

1. Jelentkezzen be az aktív HMaster ZK-ba.
1. Töltsön le egy parancsfájlt a replikáció engedélyezéséhez a következő paranccsal:
    ```
    sudo wget https://raw.githubusercontent.com/Azure/hbase-utils/master/replication/hdi_enable_replication.sh
    ```
1. Írja be a `sudo kinit -k -t /etc/security/keytabs/hbase.service.keytab hbase/<FQDN>@<DOMAIN>` parancsot.
1. Írja be a következő parancsot:

    ```bash
    sudo bash hdi_enable_replication.sh -s <srclusterdns> -d <dstclusterdns> -sp <srcclusterpasswd> -dp <dstclusterpasswd> -copydata
    ```

#### <a name="phoenix-sqlline-stops-working-after-migrating-hbase-cluster-to-hdinsight-40"></a>A Phoenix Sqlline leáll, miután a HBase fürtöt áttelepítette a HDInsight 4.0-s csomagra

Tegye a következő lépéseket:

1. Dobd el a következő Phoenix táblázatokat:
    1. `SYSTEM.FUNCTION`
    1. `SYSTEM.SEQUENCE`
    1. `SYSTEM.STATS`
    1. `SYSTEM.MUTEX`
    1. `SYSTEM.CATALOG`
1. Ha egyik tábla sem törölhető, indítsa újra a HBase billentyűt a táblákkal létesített kapcsolatok törléséhez.
1. Futtassa ismét az `sqlline.py` parancsot. A Phoenix újra létrehozza az 1.
1. A HBase-adatok Phoenix-tábláinak és nézeteinek újragenerálása.

#### <a name="phoenix-sqlline-stops-working-after-replicating-hbase-phoenix-metadata-from-hdinsight-36-to-40"></a>A Phoenix Sqlline leáll, miután replikálja a HBase Phoenix metaadatait a HDInsight 3.6-ról 4.0-ra

#### <a name="mahout"></a>Mahout

A HDP-2.3.x és a 2.4.x-ben a Mahout adott Apache-kiadásának szállítása helyett az Apache Mahout egy adott felülvizsgálati pontjával szinkronizáltuk. Ez a felülvizsgálati pont a 0.9.0-s kiadás után, de a 0.10.0-s kiadás előtt van. Ez számos hibajavítást és funkcionális fejlesztést biztosít a 0.9.0-s kiadáson keresztül, de stabil kiadást biztosít a Mahout funkciónak, mielőtt a teljes átalakítás az új Spark-alapú Mahout-ra 0.10.0-ban.

A Mahout számára a HDP 2.3.x és a 2.4.x verziópont az Apache Mahout "mahout-0.10.x" ágából származik, 2014.

A HDP-2.5.x és a 2.6.x-ben eltávolítottuk a "commons-httpclient" könyvtárat a Mahoutból, mert elavult könyvtárként tekintünk rá a lehetséges biztonsági problémákkal, és frissítettük a Hadoop-Client-t Mahout-ban a 2.7.3-as verzióra, a HDP-2.5-ben használt verzióra. Ennek eredményeként:

-   A korábban összeállított Mahout-feladatokat újra kell fordítani a HDP-2.5 vagy 2.6 környezetben.

-   Kis esély van arra, hogy egyes Mahout-feladatok "ClassNotFoundException" vagy "Could Could load class" hibákkal találkozhatnak az "org.apache.commons.httpclient", a "net.java.dev.jets3t" vagy a kapcsolódó osztálynév-előtagok hoz. Ha ezek a hibák előfordulnak, érdemes megfontolnia, hogy manuálisan telepítse-e a szükséges üvegeket a feladathoz szükséges üvegekbe, ha az elavult könyvtárbiztonsági problémáinak kockázata elfogadható a környezetben.

-   Van egy még kisebb lehetőség, hogy egyes Mahout-feladatok találkozhatnak összeomlik a Mahout hbase-ügyfél kód hívások a hadoop-közös könyvtárak, bináris kompatibilitási problémák miatt. Sajnálatos módon, nincs mód-hoz határozat ez kérdés kivéve visszatér a HDP-2.4.2 változata Mahout, amely lehet, hogy a biztonsági problémák.   -   [RANGER-1607](https://issues.apache.org/jira/browse/RANGER-1607): Megoldás a Ranger szolgáltatás ellenőrzésének hibájára, miközben a korábbi HDP-verziók HDP 2.6.2-re frissített.

      > [!NOTE]  
      > Csak akkor, ha a Ranger SSL engedélyezve van. 
     
  Ez a probléma akkor merül fel, amikor a korábbi HDP-verziókhdp-2.6.1-re való frissítésre próbál nak az Ambari-n keresztül. Ambari curl hívást használ, hogy egy szolgáltatás ellenőrzést ranger szolgáltatás Ambari. Ha az Ambari által használt JDK-verzió JDK-1.7, a curl hívás sikertelen lesz az alábbi hibával:
    
  `curl: (35) error:14077410:SSL routines:SSL23_GET_SERVER_HELLO:sslv3 alert handshake failure`
    
  Ennek oka a hiba a tomcat használt változat Ranger a Tomcat-7.0.7\*. A JDK-1.7 használata ütközik a Tomcat-7.0.7\*alapértelmezett titkosításaival.
    
  A probléma kétféleképpen oldható meg:
    
  -   Frissítse az Ambariban használt JDK-t JDK-1.7-ről JDK-1.8-ra (lásd [a JDK-verzió módosítása](https://docs.hortonworks.com/HDPDocuments/Ambari-2.2.0.0/bk_ambari_reference_guide/content/ch_changing_the_jdk_version_on_an_existing_cluster.html) az Ambari referenciaútmutatóban című szakaszt).
    
  -   Ha továbbra is támogatni szeretné a JDK-1.7 környezetet:
    
      1.  Adja hozzá a property ranger.tomcat.ciphers a ranger-admin-site szakaszban az Ambari Ranger konfiguráció az alábbi érték:
            
          SSL\_\_RSA\_RC4\_\_128 MD5,\_SSL\_RSA RC4\_\_\_128\_SHA, TLS\_\_RSA\_\_\_\_\_\_\_AES\_128 CBC SHA,\_SSL RSA 3DES EDE CBC SHA-val
        
      2.  Ha a környezet ranger-KMS-re van konfigurálva, adja hozzá a property ranger.tomcat.ciphers fájlt az Ambari Ranger konfigurációban a theranger-kms-ben az alábbi értékkel:
            
          SSL\_\_RSA\_RC4\_\_128 MD5,\_SSL\_RSA RC4\_\_\_128\_SHA, TLS\_\_RSA\_\_\_\_\_\_\_AES\_128 CBC SHA,\_SSL RSA 3DES EDE CBC SHA-val
    
  >[!NOTE]  
  >A jelzett értékek működő példák, és nem feltétlenül jelzik a környezetet. Győződjön meg arról, hogy a tulajdonságok beállításának módja megegyezik a környezet konfigurálásával.   

- **RangerUI: Escape of policy condition text entered in the policy form**
    
  **Érintett összetevő:** Ranger
    
  **A probléma leírása**
    
  Ha egy felhasználó egyéni házirendfeltételekkel szeretne házirendet létrehozni, és a kifejezés vagy a szöveg speciális karaktereket tartalmaz, akkor a házirend-kényszerítés nem fog működni. A speciális karakterek a házirend adatbázisba mentése előtt ASCII-vé alakulnak.
    
  **Különleges karakterek:**  &  &lt; &gt; " " \`

### <a name="deprecation"></a>Elavulás

-   **OMS portál:** Eltávolítottuk a hivatkozást a HDInsight erőforráslapról, amely az OMS portálra mutatott. Az Azure Monitor naplói kezdetben az OMS-portál nevű saját portálját használták a konfiguráció kezeléséhez és az összegyűjtött adatok elemzéséhez. A portál összes funkciója átlett helyezve az Azure Portalra, ahol tovább fejleszthető. A HDInsight elavulttá tette az OMS-portál támogatását. Az ügyfelek a HDInsight Azure Monitor naplóinak integrációját fogják használni az Azure Portalon.

### <a name="upgrading"></a>Frissítés

Ezek a funkciók a HDInsight 3.6-ban érhetők el. A Spark, a Kafka és az R Server (Machine Learning Services) legújabb verziójának betöltéséhez válassza a Spark, Kafka, ML Services verziót [hdinsight 3.6-os fürt létrehozásakor.](https://docs.microsoft.com/azure/hdinsight/hdinsight-hadoop-provision-linux-clusters) Az ADLS támogatásához kiválaszthatja az ADLS-tárolótípust. A meglévő fürtök nem frissülnek automatikusan ezekre a verziókra.

A 2018 júniusa után létrehozott összes új fürt automatikusan megkapja az 1000+ hibajavítást az összes nyílt forráskódú projektben. Kövesse [ezt az](https://docs.microsoft.com/azure/hdinsight/hdinsight-upgrade-cluster) útmutatót az újabb HDInsight-verzióra való frissítéssel kapcsolatos gyakorlati tanácsokért.
