# <a name="compare-active-directory-to-azure-active-directory"></a>Az Active Directory és az Azure Active Directory összehasonlítása

Az Azure Active Directory a felhő identitás- és hozzáférés-kezelési megoldásainak következő fejlődése. A Microsoft a Windows 2000 rendszerben bevezette az Active Directory tartományi szolgáltatásokat, hogy a szervezetek felhasználónként egyetlen identitás használatával kezelhetjék a helyszíni infrastruktúra több összetevőjét és rendszerét.

Az Azure AD ezt a megközelítést a következő szintre emeli azáltal, hogy a szervezetek számára egy identitás-szolgáltatás (IDaaS) megoldást biztosít az összes felhőbeli és helyszíni alkalmazásukhoz.

A legtöbb rendszergazda ismeri az Active Directory tartományi szolgáltatások kalkulát. Az alábbi táblázat ismerteti az Active Directory-fogalmak és az Azure Active Directory közötti különbségeket és hasonlóságokat.

|Fogalom|Active Directory (AD)|Azure Active Directory |
|:-|:-|:-|
|**Felhasználók**|||
|Kiépítés: felhasználók | A szervezetek manuálisan hoznak létre belső felhasználókat, vagy egy házon belüli vagy automatizált kiépítési rendszert , például a Microsoft Identity Managert használnak a HR-rendszerrel való integrációhoz.|A meglévő AD-szervezetek [az Azure AD Connect](https://docs.microsoft.com/azure/active-directory/hybrid/how-to-connect-sync-whatis) használatával szinkronizálják az identitásokat a felhőbe.</br> Az Azure AD támogatást nyújt a felhasználók automatikus létrehozásához [a felhőbeli HR-rendszerekből.](https://docs.microsoft.com/azure/active-directory/saas-apps/workday-tutorial) </br>Az Azure AD az [SCIM-kompatibilis](https://docs.microsoft.com/azure/active-directory/manage-apps/use-scim-to-provision-users-and-groups) SaaS-alkalmazásokban is kiépíthetidentitásokat, hogy automatikusan biztosítsák az alkalmazások számára a felhasználók számára a hozzáférés engedélyezéséhez szükséges adatokat. |
|Kiépítés: külső identitások| A szervezetek manuálisan hoznak létre külső felhasználókat egy dedikált külső AD-erdőben, ami felügyeleti többletterhelést eredményez a külső identitások (vendégfelhasználók) életciklusának kezeléséhez.| Az Azure AD egy speciális identitásosztályt biztosít a külső identitások támogatásához. [Az Azure AD B2B](https://docs.microsoft.com/azure/active-directory/b2b/) kezeli a külső felhasználói identitásra mutató hivatkozást, hogy megbizonyosodjon arról, hogy azok érvényesek. |
| Jogosultságkezelés és csoportok| A rendszergazdák a felhasználókat csoportok tagjává teszik. Az alkalmazás- és erőforrás-tulajdonosok ezután hozzáférést biztosítanak a csoportoknak az alkalmazásokhoz vagy az erőforrásokhoz.| [Csoportok](https://docs.microsoft.com/azure/active-directory/fundamentals/active-directory-groups-create-azure-portal) is elérhetők az Azure AD-ben, és a rendszergazdák is használhatnak csoportokat az erőforrások engedélyek megadásához. Az Azure AD-ben a rendszergazdák manuálisan rendelhetnek tagságot a csoportokhoz, vagy lekérdezéssel dinamikusan bevonhatják a felhasználókat egy csoportba. </br> A rendszergazdák az Azure AD [jogosultságkezelés](https://docs.microsoft.com/azure/active-directory/governance/entitlement-management-overview) használatával biztosíthatják a felhasználók számára az alkalmazások és erőforrások munkafolyamatok és szükség esetén időalapú feltételek használatával történő elérését. |
| Rendszergazdai felügyelet|A szervezetek tartományok, szervezeti egységek és csoportok kombinációját használják az AD-ben a felügyeleti jogok delegálására az általa ellenőrzött címtár és erőforrások kezeléséhez.| Az Azure AD [beépített szerepköröket](https://docs.microsoft.com/azure/active-directory/fundamentals/active-directory-users-assign-role-azure-portal) biztosít a szerepköralapú hozzáférés-vezérlési (RBAC) rendszer, korlátozott támogatással [egyéni szerepkörök létrehozásához](https://docs.microsoft.com/azure/active-directory/users-groups-roles/roles-custom-overview) az identitásrendszer, az alkalmazások és az általa ellenőrzött erőforrások bizalmas hozzáférésének delegálásához.</br>A szerepkörök kezelése emelt [szintű identitáskezeléssel (PIM)](https://docs.microsoft.com/azure/active-directory/privileged-identity-management/pim-configure) javítható, így csak időben, időben korlátozott vagy munkafolyamat-alapú hozzáférést biztosít a kiemelt szerepkörökhöz. |
| Hitelesítő adatok kezelése| Az Active Directoryban lévő hitelesítő adatok jelszavakon, tanúsítvány-hitelesítésen és intelligenskártyás hitelesítésen alapulnak. A jelszavak kezelése a jelszó hosszán, lejáratán és összetettségén alapuló jelszóházirendekkel van kezelve.|Az Azure AD intelligens [jelszavas védelmet](https://docs.microsoft.com/azure/active-directory/authentication/concept-password-ban-bad) használ a felhőben és a helyszíni használatra. A védelem intelligens zárolást, valamint a gyakori és egyéni jelszókifejezések és helyettesítések blokkolását foglalja magában. </br>Az Azure AD jelentősen növeli a biztonságot [a többtényezős hitelesítés](https://docs.microsoft.com/azure/active-directory/authentication/concept-mfa-howitworks) és a [jelszó nélküli](https://docs.microsoft.com/azure/active-directory/authentication/concept-authentication-passwordless) technológiák, például a FIDO2. </br>Az Azure AD csökkenti a támogatási költségeket azáltal, hogy [önkiszolgáló jelszó-visszaállítási](https://docs.microsoft.com/azure/active-directory/authentication/concept-sspr-howitworks) rendszert biztosít a felhasználóknak. |
| **Alkalmazások**|||
| Infrastruktúra-alkalmazások|Az Active Directory számos helyszíni infrastruktúra-összetevő alapját képezi, például a DNS, a DHCP, az IPSec, a WiFi, a NPS és a VPN-hozzáférés.|Egy új felhővilág, az Azure AD, az új vezérlősík az alkalmazások eléréséhez, szemben támaszkodva hálózati vezérlők. Amikor a felhasználók hitelesítik[, a feltételes hozzáférés (CA)](https://docs.microsoft.com/azure/active-directory/conditional-access/overview)szabályozza, hogy mely felhasználók férhetnek hozzá a szükséges feltételek mellett lévő alkalmazásokhoz.|
| Hagyományos és régebbi alkalmazások| A legtöbb helyszíni alkalmazás LDAP, Windows-integrált hitelesítés (NTLM és Kerberos) vagy fejlécalapú hitelesítést használ a felhasználók hozzáférésének szabályozásához.| Az Azure AD hozzáférést biztosíthat az ilyen típusú helyszíni alkalmazásokhoz a helyszíni Azure [AD-alkalmazás proxyügynökök](https://docs.microsoft.com/azure/active-directory/manage-apps/application-proxy) használatával. Ezzel a módszerrel az Azure AD hitelesítheti az Active Directory-felhasználókat a kerberos használatával, miközben áttelepíti, vagy együtt kell léteznie az örökölt alkalmazásokkal. |
| SaaS-alkalmazások|Az Active Directory natívmódon nem támogatja az SaaS-alkalmazásokat, és összevonási rendszert igényel, például az AD FS-t.|Az OAuth2, SAML és WS-hitelesítést\* támogató SaaS-alkalmazások integrálhatók az Azure AD hitelesítéshez való használatához. |
| Üzletági (LOB) alkalmazások modern hitelesítéssel|A szervezetek az Active Directoryval rendelkező AD FS segítségével támogathatják a modern hitelesítést igénylő üzletági alkalmazásokat.| A modern hitelesítést igénylő LOB-alkalmazások konfigurálhatók úgy, hogy az Azure AD-t használják a hitelesítéshez. |
| Középső szintű/démonszolgáltatások|A helyszíni környezetekben futó szolgáltatások általában AD szolgáltatásfiókokat vagy csoportfelügyelt szolgáltatásfiókokat (gMSA) használnak a futtatáshoz. Ezek az alkalmazások ezután öröklik a szolgáltatásfiók engedélyeit.| Az Azure AD [felügyelt identitásokat](https://docs.microsoft.com/azure/active-directory/managed-identities-azure-resources/index) biztosít a felhőben más számítási feladatok futtatásához. Ezek az identitások életciklusa az Azure AD által kezelt, és az erőforrás-szolgáltatóhoz van kötve, nem használható más célokra a backdoor hozzáférés megszerzéséhez.|
| **Eszközök**|||
| Mobiltelefon|Az Active Directory nem támogatja natív módon a mobileszközöket külső megoldások nélkül.| A Microsoft mobileszköz-felügyeleti megoldása, a Microsoft Intune integrálva van az Azure AD-vel. A Microsoft Intune eszközállapot-adatokat biztosít az identitásrendszernek a hitelesítés során kiértékelhető módon. |
| Windows-asztalok|Az Active Directory lehetővé teszi a Windows-eszközökhöz való csatlakozást a Csoportházirend, a System Center Configuration Manager vagy más külső megoldások használatával.|A Windows-eszközök [csatlakoztathatók az Azure AD szolgáltatáshoz.](https://docs.microsoft.com/azure/active-directory/devices/) Feltételes hozzáférés ellenőrizheti, ha egy eszköz az Azure AD csatlakozott a hitelesítési folyamat részeként. A Windows-eszközök a [Microsoft Intune segítségével](https://docs.microsoft.com/intune/what-is-intune)is kezelhetők. Ebben az esetben a feltételes hozzáférés, megvizsgálja, hogy egy eszköz panasz (például naprakész biztonsági javítások és vírusaláírások), mielőtt hozzáférést biztosít az alkalmazásokhoz.|
| Windows-kiszolgálók| Az Active Directory erős felügyeleti lehetőségeket biztosít a helyszíni Windows-kiszolgálók számára csoportházirend vagy más felügyeleti megoldások használatával.| Az Azure-ban található Windows-kiszolgálók virtuális gépei kezelhetők az [Azure AD tartományi szolgáltatásokkal.](https://docs.microsoft.com/azure/active-directory-domain-services/) [Felügyelt identitások](https://docs.microsoft.com/azure/active-directory/managed-identities-azure-resources/index) akkor használhatók, ha a virtuális gépeknek hozzáférésre van szükségük az identitásrendszer-címtárhoz vagy -erőforrásokhoz.|
| Linux/Unix számítási feladatok|Az Active Directory natív módon nem támogatja a nem Windows rendszert külső megoldások nélkül, bár a Linux-gépek konfigurálhatók úgy, hogy az Active Directoryval Kerberos-rendszerként hitelesítsék magukat.|A Linux/Unix virtuális gépek [felügyelt identitások](https://docs.microsoft.com/azure/active-directory/managed-identities-azure-resources/index) használatával érhetik el az identitásrendszert vagy az erőforrásokat. Egyes szervezetek ezeket a számítási feladatokat felhőbeli tárolótechnológiákba telepítheti át, amelyek felügyelt identitásokat is használhatnak.|
